1. What the Project Does

The one-liner A user asks: "I am spending ₹50,000 on flights. I have Axis Atlas, HDFC DCB, and SBI Cashback. Which card should I use?" — and the system gives an explainable, calculated, grounded answer. Not from memory. From retrieved documents.

The system must combine three abilities the PDF emphasises:

AbilityWhy it's hardHow we solve it
Retrieve the correct ruleFlight rule ≠ hotel rule ≠ rent rule; different per cardHybrid RAG + SQL lookup from reward_rules table
Reason with numbersLLMs make arithmetic errors; caps / exclusions varyDeterministic calculator tool — LLM never does math
Avoid hallucinationCard terms change; LLM may invent ratesEvidence gate + post-generation guardrail LLM check

5. System Architecture — 5 Layers

Layer 1 — Data Ingestion

PDF / Markdown file │ ▼ extract_pages() ← rag/ingest_pdfs.py (PyMuPDF) │ ▼ chunk_text() ← rag/ingest_pdfs.py (900 chars, 150 overlap) │ ▼ embed_texts() ← rag/embeddings.py (fastembed or hashing) │ ▼ INSERT document_chunks ← pgvector IVFFlat index built automatically │ ▼ extract_card() ← rag/extract_rules.py (LLM extracts structured rules) │ ▼ INSERT reward_rules ← 55 rules for 5 cards currently in DB

Entry point: seed.py → run with make seed. Dynamic add from UI: rag/add_card.py

Layer 2 — Retrieval (Hybrid)

User query │ ▼ embed_query() ← rag/embeddings.py │ ▼ cosine distance SQL ← pgvector SELECT ... ORDER BY embedding <=> $1 │ ▼ keyword boost +0.03 per keyword hit → re-ranked │ ▼ top-K chunks default top_k = 6 Parallel SQL lookups also run in retrieve_node: fetch_rule() ← tools/retriever.py (reward_rules per card+category) fetch_transfer_partners() ← tools/retriever.py (transfer_partners table)

Layer 3 — LangGraph Agent Reasoning

START │ ▼ [memory_load] ← loads user profile from DB │ ▼ [classify] ← LLM parses intent, spend items, card names, safety flags │ ├─ off-topic / injection ──► [refusal] ──► END ├─ needs_clarification ──► [clarify] ──► (interrupt — waits for user) ──► [retrieve] └─ on-topic ──► [retrieve] │ [validate] ← rule_validator.py │ ├─ invalid ──► [insufficient] ──► END └─ valid ──► [compute] ← calculator.py called here │ ├─ transfer intent ──► [approval] (interrupt) │ ├─ yes ──► [generate] │ └─ no ──► [cancelled] ──► END └─ other ──────────► [generate] │ [guardrail] │ [persist] ← upserts user profile │ END

Layer 4 — Tool Use (No LLM Math)

PDF requirement "The agent should NOT do calculations inside the LLM response. It should call a calculator tool."

Calculator tool — tools/calculator.py → compute_reward()

# Input (from reward_rules DB row):
spend            = 50000
reward_rate      = 5          # points
reward_per_amount = 100       # per ₹100
point_value      = 1.0        # ₹1 per point
monthly_cap      = 5000       # max points

# Output — fully deterministic:
units_earned      = 2500
reward_value_rs   = 2500.0
cap_applied       = False
excluded          = False
effective_return  = 5.0%      # reward_value / spend * 100

Transfer calculator — tools/transfer_calculator.py → compare_transfers() ranks all partners by output value.

Evidence gate — tools/rule_validator.py → validate_evidence() hard-stops if no chunk scores above MIN_SIMILARITY=0.30 — prevents guessing.

8. Human-in-the-Loop (HITL)

PDF specifies two HITL scenarios. Both use LangGraph's interrupt() mechanism.

Scenario 1: Clarification
agents/nodes.py → clarify_node

When intent is clear but preferences are missing. Graph pauses, Streamlit shows a text input. User types answer. resume_agent("hotel points") called. Graph continues from where it stopped.

Scenario 2: Transfer Approval
agents/nodes.py → approval_node

Before any point transfer recommendation. Graph literally cannot route to generate without an explicit yes/no. Streamlit shows green Approve and red Cancel buttons.

Both interrupts are surfaced via interrupted, needs_clarification, needs_approval flags in AgentState. Resume via POST /resume API or resume_agent() in agents/graph.py.
